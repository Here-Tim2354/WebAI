# WebAI Blueprint

## 项目定位
WebAI 是一个基于 Supabase 的 WebAI 对话与会话持久化管理系统。

它既是一个面向用户的 AI 聊天产品原型，也是一个面向数据库课程设计的完整应用系统。
项目将以“自然聊天体验 + 清晰数据库设计 + 可展示的会话管理能力”为核心方向推进。

默认将本项目理解为：
- 一个以 AI 对话和会话管理为核心的 Web 产品
- 一个以前后端统一 TypeScript 为主方向的实现方案
- 一个既服务产品体验、又服务课程设计交付的长期项目

## 产品目标
优先保证以下几点：
- 聊天体验自然、稳定、现代
- 页面简洁清晰，具备现代感和可演示性
- 会话可以保存、恢复和管理
- 前后端边界统一，减少跨端数据契约混乱
- 产品既能用于演示，也能支撑课程设计答辩

## 课程设计目标
本项目需要满足数据库课程设计的关键要求：
- 必须使用数据库存储核心数据
- 用户界面需要体现增删查改能力
- 数据库设计应能从用户需求推导出来
- 系统需要具备需求分析、数据库设计、实现与展示的完整闭环

因此，本项目不只追求“能聊天”，还要体现：
- 会话与消息数据模型
- 以用户视角组织的数据对象
- 数据持久化与管理能力
- 面向答辩的可解释性与可展示性

## 预期技术路线
- 前后端统一采用 TypeScript
- Web 框架采用 Next.js App Router
- 前端负责聊天界面、会话管理界面与交互体验
- 服务端负责 AI 调用、数据写入与业务编排
- Supabase 作为核心持久化数据库
- 跨端数据校验采用 Zod
- Gemini 首版继续放在 Next 服务端接入
- 采用 Figma MCP 进行用户与 AI 协作开发

## 核心业务对象
围绕以下对象展开系统设计：
- user
- conversation
- message
- 可扩展对象：prompt template / tag / favorite / attachment metadata

默认核心主线为：
- 用户进入系统
- 创建或打开会话
- 发送消息并获得 AI 回复
- 历史消息写入数据库
- 用户可查看、管理和恢复会话

## Phase 路线

### Phase 1：聊天主链路 MVP
目标：先做出一个可用、好看、结构正确的聊天网页。

包括：
- 聊天页面
- 消息列表
- 输入框与发送按钮
- 加载状态与错误反馈
- Gemini 非流式调用
- 单页内多轮对话

完成标志：
- 用户能发送消息并收到 Gemini 回复
- 页面可以连续对话
- API Key 不暴露给客户端

### Phase 2：体验优化与结构整理
目标：在不引入重度返工的前提下，优化聊天体验并整理前端结构，为数据库接入做准备。

包括：
- 输入体验优化
- 自动滚动优化
- 更清晰的消息状态反馈
- 组件边界整理
- 前后端数据结构进一步统一

完成标志：
- 聊天交互更顺手
- 代码结构更适合承接持久化和会话系统

### Phase 3：Supabase 持久化与会话 CRUD 系统
目标：完成课程设计所需的数据库主线，让会话与消息真正进入数据库。

包括：
- 接入 Supabase
- 建立 users / conversations / messages 等核心数据模型
- 新建会话
- 查看会话列表
- 恢复历史会话
- 重命名会话
- 删除会话
- 消息持久化与读取

完成标志：
- 会话数据可持久化
- 用户界面体现完整的会话管理能力
- 系统具备明确的数据库支撑价值

### Phase 4：流式输出与生成控制
目标：在数据库主线稳定后，补齐更像 AI 产品的生成体验。

包括：
- 流式输出
- 增量渲染 assistant 回复
- 中断生成
- 更细粒度的消息状态管理

完成标志：
- 回复可以逐步显示
- 用户能感知更快反馈
- 生成过程更可控

### Phase 5：AI 能力扩展与课程答辩打磨
目标：在主链路和数据库能力稳定后，补充可展示亮点并整理课程交付材料。

包括：
- 可扩展 AI 能力预留
- 文档与数据库设计对齐
- 安装说明与演示路径整理
- 答辩展示优化

完成标志：
- 产品可演示
- 文档、代码、答辩三者叙事一致
- 项目具备完整课程设计交付能力

## 当前阶段判断
当前项目已进入 `Phase 3`，并已完成 `Phase 3.3` 的 Supabase 接入层初始化。

下一步应继续推进 `Phase 3.4` 的会话级 CRUD，让数据库主线尽快从“已接入”走向“可演示、可管理、可恢复”。

## 设计边界与取舍
- 浏览器只调用 Next 提供的服务端接口
- Gemini API Key 仅存在于服务端
- 所有跨端数据都应通过统一 schema 校验
- 数据库中的核心对象优先围绕会话与消息展开
- 优先完成数据库课程设计所需的主线能力
- 不优先投入复杂用户系统
- 不提前做过多高阶 AI 扩展
- 产品体验与数据库价值要同时可见，但优先保证主链路可交付
