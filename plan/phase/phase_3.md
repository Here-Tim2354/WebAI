# Phase 3

## 目标
- 以 `Supabase PostgreSQL` 为核心，完成会话与消息的真实持久化
- 在用户界面上形成可演示、可答辩的会话级 CRUD 闭环
- 将当前项目从“单页单会话前端原型”推进为“有数据库主线支撑的聊天系统”
- 同时服务产品主链路与数据库课程设计要求，建立 `需求 -> 数据字典 -> ER/关系模式 -> migration -> CRUD 实现` 的完整闭环

## 本阶段定位
- `Phase 3` 是当前项目的数据库主战场，也是课程设计实现阶段的核心
- 本阶段重点不是“用了 Supabase”，而是“把数据库设计真正落到系统主链路中”
- 本阶段默认优先保证：
  - 会话 CRUD 可真实演示
  - 消息写入数据库
  - 历史会话可恢复
  - 数据库设计能从需求文档反推出并解释清楚

## 前置原则
- 任何 `Phase 3` 的实现、调整和方案判断，都不应脱离课程文档与官方资料单独拍脑袋推进
- 在开始建表、接接口、改页面前，应先完成两类前置确认：
  - 文档前置确认
  - `Supabase MCP / 官方文档` 前置确认
- 若前置确认未完成，不应直接进入大规模实现

## 本阶段结论
- 当前 `Phase 2` 的前端体验收口已基本完成，可以把主线切向 `Phase 3`
- `Phase 3` 不应从“直接改聊天接口”开始，而应先收口数据库口径、完成核心表设计，再逐步接入页面
- 本阶段不追求一次性做完整的 AI 产品能力扩展，而要优先守住课程要求中的数据库 CRUD 主线

## 产品边界
- 本阶段必须完成：
  - 会话持久化
  - 消息持久化
  - 会话列表查询
  - 历史会话恢复
  - 会话重命名
  - 会话删除
- 本阶段建议完成：
  - `Supabase Auth` 与用户归属关系接入
  - 基础 `RLS`
  - 关键索引、约束、删除策略
- 本阶段暂不作为主任务：
  - 完整流式输出
  - 中断生成
  - 标签系统
  - 收藏系统全量实现
  - 正式搜索系统
  - 附件上传
  - 管理员后台完整落地

## 课程设计导向
- 本阶段所有实现都要围绕课程要求中的几个重点展开：
  - 必须使用数据库存储核心数据
  - 用户界面必须体现增删查改
  - 数据字典必须从用户视角出发，而不是数据库字段表
  - 数据库设计需要能解释实体、联系、主键、外键、约束、索引
  - 系统实现要能说明数据库连接与使用方式
- 对外表达时，应将 Supabase 定位为：
  - `Postgres 数据库 + Auth + Storage + 平台能力`
  - 而不是课程答辩的唯一主角

## 文档前置工作
- 在 `Phase 3` 中，所有数据库实现都应先回到课程与项目文档确认口径
- 默认需要优先阅读并对齐：
  - `requirements/scau_database_course_design_ai_readable.md`
  - `requirements/supabase_course_design_checklist.md`
  - `plan/phase/current_todo.md`
  - `plan/database/` 下的需求分析、数据字典、实体关系、关系模式等文档
- 前置确认重点包括：
  - 当前实现是否仍然服务课程的数据库主线
  - 数据字典是否仍然从用户视角出发，而不是退化成字段表
  - 当前新增功能是否真的支撑 CRUD 演示
  - 当前数据库结构是否仍然能从需求和用例反推
- 若项目文档与实现意图发生冲突，应优先回到文档口径重新判断，而不是直接修改数据库结构

## Supabase 查询前置工作
- `Phase 3` 的实现细节不应仅依赖记忆，应优先通过官方资料确认最新口径
- 默认查询顺序建议为：
  1. 先看项目内文档
  2. 再查 `Supabase MCP`
  3. 若 `MCP` 返回异常、缺失、超时或结果不稳定，再回退到 Supabase 官方文档
- 需要优先确认的内容包括：
  - 当前 Free 计划限制
  - `Supabase Auth` 与 `profiles` 的推荐职责划分
  - `Next.js` 下浏览器端与服务端 client 的推荐接入方式
  - `RLS`、migration、函数、Storage 的官方推荐做法
  - 若使用特定平台能力，其限制和官方约束是否有变化

## Supabase MCP 状态确认
- 当前项目应默认认为：`Supabase MCP` 是重要辅助工具，但不是绝对稳定、绝对可靠的单一事实来源
- 在每次进入 `Phase 3` 的关键实现前，建议先进行一次轻量状态确认：
  - 文档检索是否可正常返回
  - 数据库工具是否可调用
  - 返回内容是否完整且与当前任务相关
- 若发现以下情况，应立即触发降级策略：
  - 无法正常返回结果
  - 返回内容明显缺失
  - 多次查询结果不一致
  - 工具调用偶发失败，影响方案判断
- 降级策略：
  - 以 Supabase 官方文档为主
  - 再结合项目已有数据库文档和本地实现判断
  - 避免把关键设计建立在一次不稳定的 MCP 返回之上
- 对需要强时效性的内容，应在实施前再次确认，而不是复用过旧结论

## 核心推进策略
- 先收口数据库与课程文档口径，再落数据库结构
- 先确认 `Supabase MCP` 与官方资料口径，再决定具体实现细节
- 先完成会话系统，再改聊天主链路
- 先保证可演示的数据库 CRUD，再补权限和加分项
- 让每一步产出都能同时服务：
  - 项目实现
  - 课程文档
  - 后续答辩展示

## 推荐实现顺序
1. 先阅读课程要求、执行清单和 `plan/database/` 文档，统一数据库表达口径
2. 先确认 `Supabase MCP` 当前状态，并通过官方资料核实本轮实现依赖的关键能力
3. 完成 `Supabase` 核心关系模式的 migration 设计与落地
4. 接入浏览器端与服务端 `Supabase client`，明确身份获取与数据库访问边界
5. 先实现会话级 CRUD：新建、查询、打开、重命名、删除
6. 再将聊天主链路改为数据库驱动：写入用户消息、生成 AI 回复、写入 assistant 消息、恢复历史消息
7. 最后补基础 `RLS`、关键索引验证、文档映射和答辩表达材料

## 分阶段安排

### 3.1 数据库口径收口
- 目标：
  - 让数据库文档真正成为实现依据，而不是独立的说明材料
- 重点：
  - 先回读课程要求与执行清单
  - 复查用户视角数据字典是否符合课程要求
  - 收敛核心实体，优先固定为：
    - `auth.users / profiles`
    - `conversations`
    - `messages`
  - 明确首批必做与后续扩展的边界
- 产出：
  - 可直接指导建表与实现的数据库文档口径

### 3.2 核心表与 migration 落地
- 目标：
  - 把关系模式真正落到 `Supabase PostgreSQL`
- 重点：
  - 在落地前先确认本轮依赖的 `Supabase MCP` 与官方文档口径
  - 建立 `profiles`、`conversations`、`messages`
  - 明确主键、外键、可空性、默认值
  - 配置关键索引
  - 设计删除策略与更新时间维护策略
  - 视实现成本加入基础触发器或函数
- 产出：
  - 首批 migration
  - 可解释的数据库实现基础

### 3.3 接入层建立
- 目标：
  - 建立统一、清晰的数据库访问方式
- 重点：
  - 优先确认 `Next.js + Supabase` 在当前官方文档中的推荐接法
  - 浏览器端 `Supabase client`
  - 服务端 `Supabase client`
  - 环境变量与密钥边界
  - 当前用户身份获取方式
  - `Next.js Route Handler` 中的数据库访问路径
- 产出：
  - 稳定的 Supabase 接入基础设施

### 3.4 会话 CRUD 闭环
- 目标：
  - 优先完成课程最看重、最易演示的数据库 CRUD 主线
- 重点：
  - 新建会话
  - 查询会话列表
  - 打开历史会话
  - 重命名会话
  - 删除会话
- 说明：
  - 这一阶段优先于聊天持久化改造
  - 先把“数据库里的多会话系统”立住，再改聊天主链路
- 产出：
  - 可在界面上真实演示的会话级 CRUD

### 3.5 聊天持久化闭环
- 目标：
  - 让当前聊天能力真正建立在数据库之上
- 重点：
  - 当前未打开会话时，首次发送自动创建会话
  - 用户消息先写入数据库
  - 服务端调用 Gemini 生成回复
  - assistant 回复写入数据库
  - 打开历史会话时可恢复完整消息记录
- 说明：
  - 这一阶段是本阶段耦合度最高的部分
  - 必须建立在会话 CRUD 已稳定的基础上推进
- 产出：
  - 数据库驱动的聊天主链路

### 3.6 加分项与答辩支撑材料
- 目标：
  - 在主链路稳定后，补齐课程表达与工程佐证
- 重点：
  - 基础 `RLS` 验证
  - 记录本阶段关键 Supabase 查询结论与依据来源
  - migration 与表设计说明的整理
  - 页面功能与数据库操作映射关系整理
  - 数据库连接方式、约束、索引、删除策略的文档化
- 说明：
  - 这一步是加分项，不应抢占主线实现顺序
- 产出：
  - 更完整的课程设计表达材料

## 推荐的最小数据库范围
- 第一批只落核心主线：
  - `profiles`
  - `conversations`
  - `messages`
- 当前不把以下能力作为首批数据库落地重点：
  - `favorites`
  - `search_records`
  - `tags`
  - `attachments`
- 原因：
  - 它们能增强展示度，但会显著放大本阶段改动面
  - 对课程最核心的 CRUD 主线不是首要前置条件

## 建议的数据职责划分
- `Supabase Auth`
  - 提供用户身份与登录态
- `profiles`
  - 承接展示资料与业务侧用户补充信息
- `conversations`
  - 承接会话标题、状态、归属关系、时间信息
- `messages`
  - 承接会话中的消息内容与消息时间
- `Next.js` 服务端
  - 负责 Gemini 调用、复杂业务编排和安全边界控制

## 风险与控制点
- 风险一：认证身份与业务归属关系不清
  - 控制方式：优先采用 `Supabase Auth + profiles` 的表达，不重复造认证主表
- 风险二：前端本地状态迁移到数据库驱动时逻辑混乱
  - 控制方式：先完成会话 CRUD，再推进聊天持久化
- 风险三：一次性并行改动过多导致系统不稳定
  - 控制方式：按里程碑拆分，避免流式输出、收藏、搜索等能力抢占主线

## 完成标志
- 用户界面能稳定演示：
  - 新建会话
  - 查看会话列表
  - 打开历史会话
  - 重命名会话
  - 删除会话
- 聊天消息真实写入数据库并可恢复
- 数据库表、索引、约束、删除策略可以被清楚解释
- migration 可以作为课程设计中的数据库实现证据
- 项目从“刷新即丢失的单页原型”进入“有数据库主线支撑的聊天系统”

## 当前阶段判断
- `Phase 3` 不适合再按“前端体验优化”的节奏推进
- 接下来所有新增能力都应优先服务数据库主线与课程答辩闭环
- 判断标准应从“页面是否更像产品”切换为：
  - 数据库主线是否成立
  - 会话 CRUD 是否完整
  - 聊天是否真正建立在持久化之上

## 一句话总结
- `Phase 3` 的核心不是“接上 Supabase”，而是把 `WebAI` 做成一个真正具备数据库课程设计主线的聊天系统：能解释、能演示、能落库、能支撑后续答辩。
