# Chat Workspace Layout Plan

## 目标
本文件用于描述 WebAI 聊天页的整体页面规划，重点是为未来的右侧或中间扩展工作区预留结构，而不是把当前产品做成重型多面板系统。

当前阶段的判断是：
- 聊天主链路仍然是核心
- 页面需要保留轻量仪表盘感
- 左侧栏可以承载分组历史会话与状态总览
- 右侧或中间扩展区应当作为未来能力的预留位，而不是今天就塞满

## 参考原则

### 来自 `frontend-skill`
- 先做构图和层级，再做组件
- 默认克制，少卡片，少装饰，少无意义边框
- 让主工作面足够明确
- 动效少而精，只保留真正改善感知的部分
- 页面整体优先于局部组件堆砌

### 来自 OpenAI 官方做法
- `Canvas` 的核心不是把聊天做重，而是把“需要编辑、修订、持续工作”的任务从普通对话里分离出来
- 这意味着页面应区分“对话推进”和“工作承载”两种职责
- OpenAI 的官方表述里，`Canvas` 是一个独立工作空间，而不是把聊天气泡继续横向堆叠
- 对你的项目来说，这个思路的启发是：未来扩展能力应该有自己的承载区域，而不是全部挤在聊天流里

## 推荐判断
当前阶段建议以 **方案 B 为主，方案 A 作为长期预留方向**。

原因是：
- 你并不是真的要做一个很成熟、很重的多工作区产品
- 当前业务弹窗承载量较少
- 页面要保持聊天产品的轻快感
- 未来如果真的需要更完整的工作台，再演进到方案 A 会更稳

也就是说：
- 今天的主张不是“先做大工作台”
- 而是“先保留扩展位，再根据业务密度决定是抽屉式还是固定栏式”

## 当前实现状态
- 当前页面已经按轻量工作区方向收口
- 左侧栏更接近轻导航，而不是控制台
- 中间聊天区仍然是绝对主视觉
- 右侧扩展区暂未实现，保留为后续结构扩展位

## 左侧栏定位

左侧栏建议承担三层信息：

### 第一层：导航
- 新建会话
- 会话分组
- 历史会话入口

### 第二层：状态总览
- 当前工作状态
- 最近使用的提示词分组
- 简短仪表盘信息

### 第三层：轻量管理
- 收藏
- 标签
- 最近活跃
- 可能的模式切换入口

设计原则：
- 左侧栏是“轻量信息中心”，不是完整后台
- 它要帮助用户快速定位，不要抢走主聊天区注意力
- 视觉上要更像侧边导航，而不是控制面板

## 中间主区定位

中间主区只做一件事：

**把聊天对话保持得足够顺、足够清楚、足够像主任务。**

这意味着：
- 消息流优先
- 输入区优先
- 生成状态优先
- 当前对话上下文优先

中间主区不应该承担过多“设置”“分组”“工具”“管理”职责。

## 右侧或中部扩展区预留策略

### 预留内容
未来可以考虑放进扩展区的内容包括：
- 当前会话摘要
- 模型或模式说明
- 少量业务操作
- 产物预览
- 附加说明
- 轻量状态卡

### 不建议放进去的内容
- 大量表单
- 很重的管理列表
- 过多导航入口
- 高频操作区
- 与聊天主链路无关的噪音信息

### 预留方式
- 先在布局上保留宽度和弹性
- 让扩展区默认可收起
- 在视觉上使用较弱的存在感
- 优先支持“按需出现”，其次才是“持续常驻”

## 原型与设计工作建议

这里建议做的是“结构原型”，不是完整高保真视觉稿。

建议至少补三张原型：
1. 空态首屏原型
2. 聊天进行中原型
3. 打开扩展区后的原型

每张原型都要回答三个问题：
- 主任务是谁
- 辅助区域是谁
- 扩展能力从哪里长出来

## 未来演进方向

### 早期
- 左侧轻导航
- 中间聊天主区
- 右侧预留但收起

### 中期
- 历史会话与分组更清楚
- 右侧开始承载少量上下文或业务面板
- 不同提示词场景开始有轻量仪表盘感

### 后期
- 只有当业务密度真的上来时，再考虑固定工作台结构
- 如果始终是轻量业务，就保持聊天主链路优先，扩展区按需出现即可

## 结论
- 这个项目不需要一开始就做成成熟工作台
- 但需要从现在开始保留扩展窗口
- 最合理的路线是：聊天主链路保持轻，布局骨架保持活，扩展能力按业务密度逐步长出来
