# 实体与联系分析

## 文档定位

本文基于当前已完成的：

- 用户需求分析
- 用户视角数据字典

继续抽取系统中的核心实体与联系，为后续 ER 图和关系模式设计提供基础。

本文保持精简，只回答三个问题：

1. 系统里有哪些核心实体
2. 哪些内容只是属性，不单独成实体
3. 各实体之间是什么联系

---

## 一、核心实体

### 1. 用户身份

用户身份用于表示系统中的实际使用者。

当前用户分为两类：

- 普通用户
- 管理员

在当前 `Supabase` 方案下，认证主身份默认由 `auth.users` 提供，而不是优先自建一张完整认证主表。
业务文档中提到的“用户”在落地时应理解为：

- `auth.users`：承接认证身份
- `profiles`：承接界面展示资料与业务补充信息

用户身份实体承担的核心作用是：

- 承载认证身份与系统角色
- 作为会话、收藏、搜索记录等数据的归属主体
- 区分普通用户与管理员权限

### 2. 用户资料

用户资料用于承载用户在系统界面中的展示信息。

用户资料实体承担的核心作用是：

- 保存头像、昵称或展示名称等资料
- 作为用户界面中个人入口的展示来源
- 与认证身份分离，便于课程设计中区分“登录主体”和“业务资料”

### 3. 会话

会话用于承载一次持续性的聊天主题，是普通用户管理历史记录的核心对象。

会话实体承担的核心作用是：

- 组织一组相关消息
- 支持重命名、删除、归档
- 保存会话级提示词
- 作为搜索结果和历史会话列表的基本单位

### 4. 消息

消息用于表示会话中的单条聊天内容。

消息实体承担的核心作用是：

- 记录用户提问和 AI 回复
- 构成会话的主要内容
- 作为收藏功能的直接对象
- 作为搜索命中的底层内容来源

### 5. 收藏

收藏用于表示用户对某条消息执行的收藏行为。

收藏实体承担的核心作用是：

- 建立“用户收藏了某条消息”的关系
- 支撑左侧收藏区域展示
- 支撑从收藏项跳转回原消息位置

当前约束为：

- 一个用户对同一条消息只能收藏一次

### 6. 搜索记录

搜索记录用于表示用户执行过的搜索行为。

搜索记录实体承担的核心作用是：

- 保存用户的搜索关键词
- 支撑后续搜索历史展示或分析
- 体现系统中的用户查询行为数据

## 二、不单独成实体的内容

以下内容当前应作为字段或状态处理，而不是单独拆表：

### 1. 用户角色

用户角色是用户实体的属性，不单独作为实体。

当前只需区分：

- 普通用户
- 管理员

### 2. 会话级提示词

会话级提示词本质上是某个会话的配置属性。

当前只是一段 Markdown 文本，因此作为会话字段处理即可，不单独拆实体。

### 3. 会话归档

归档本质上是会话状态，而不是独立业务对象。

因此应作为会话状态字段处理。

归档页面中看到的内容，本质上仍然是“状态为已归档的会话集合”。
恢复归档也只是会话状态流转，不需要额外拆分实体。

### 4. 消息收藏图标状态

图标点亮或未点亮只是收藏关系在界面中的表现，不是单独实体。

### 5. 搜索结果

搜索结果是系统根据关键词对消息或会话进行匹配后生成的展示结果，不是独立实体。

当前阶段只需要记录搜索行为，不需要把“结果集”作为独立实体存储。

## 三、实体联系分析

### 1. 用户 与 用户资料

联系说明：

- 一个用户对应一份用户资料
- 一份用户资料只属于一个用户

联系类型：

- 一对一

### 2. 用户 与 会话

联系说明：

- 一个用户可以拥有多个会话
- 一个会话只属于一个用户

联系类型：

- 一对多

### 3. 会话 与 消息

联系说明：

- 一个会话包含多条消息
- 一条消息只属于一个会话

联系类型：

- 一对多

### 4. 用户 与 收藏

联系说明：

- 一个用户可以收藏多条消息
- 一条收藏记录只属于一个用户

联系类型：

- 一对多

### 5. 消息 与 收藏

联系说明：

- 一条消息可以被其所属用户收藏
- 一条收藏记录只对应一条消息

联系类型：

- 一对多

补充说明：

- 从业务约束看，收藏本质上是用户与消息之间的关联关系
- 当前通过独立收藏实体承接，并限制“同一用户对同一消息只能收藏一次”

### 6. 用户 与 搜索记录

联系说明：

- 一个用户可以有多条搜索记录
- 一条搜索记录只属于一个用户

联系类型：

- 一对多

## 四、当前可直接形成的实体集合

基于整体需求，推荐的实体集合为：

- 用户身份
- 用户资料
- 会话
- 消息
- 收藏
- 搜索记录

其中，在 `Phase 3` 的第一批数据库落地中，应优先固定为：

- 用户身份（`auth.users`）
- 用户资料（`profiles`）
- 会话（`conversations`）
- 消息（`messages`）

收藏和搜索记录虽然来自需求，但当前不应与数据库主线同批次强绑定实现。

进一步说：

- 用户身份、用户资料、会话、消息是数据库主线
- 收藏和搜索记录是围绕主线展开的扩展实体

## 六、一句话总结

当前系统以“用户身份 - 用户资料 - 会话 - 消息”为核心主线，并通过“收藏”和“搜索记录”补充用户管理历史内容的能力；在 `Phase 3` 首批落地中，应优先实现主线实体，会话级提示词、归档状态、用户角色等内容继续作为属性处理。
