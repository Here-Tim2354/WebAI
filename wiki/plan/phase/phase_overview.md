# WebAI Blueprint

## 项目定位
WebAI 是一个基于 Supabase 的 WebAI 对话与会话持久化管理系统。

它既是一个面向用户的 AI 聊天产品原型，也是一个面向数据库课程设计的完整应用系统。
项目将以“自然聊天体验 + 清晰数据库设计 + 可展示的会话管理能力”为核心方向推进。

默认将本项目理解为：
- 一个以 AI 对话和会话管理为核心的 Web 产品
- 一个以前后端统一 TypeScript 为主方向的实现方案
- 一个既服务产品体验、又服务课程设计交付的长期项目

## 产品目标
优先保证以下几点：
- 聊天体验自然、稳定、现代
- 页面简洁清晰，具备现代感和可演示性
- 会话可以保存、恢复和管理
- 前后端边界统一，减少跨端数据契约混乱
- 产品既能用于演示，也能支撑课程设计答辩

## 课程设计目标
本项目需要满足数据库课程设计的关键要求：
- 必须使用数据库存储核心数据
- 用户界面需要体现增删查改能力
- 数据库设计应能从用户需求推导出来
- 系统需要具备需求分析、数据库设计、实现与展示的完整闭环

因此，本项目不只追求“能聊天”，还要体现：
- 会话与消息数据模型
- 以用户视角组织的数据对象
- 数据持久化与管理能力
- 面向答辩的可解释性与可展示性

## 预期技术路线
- 前后端统一采用 TypeScript
- Web 框架采用 Next.js App Router
- 前端负责聊天界面、会话管理界面与交互体验
- 服务端负责 AI 调用、数据写入与业务编排
- Supabase 作为核心持久化数据库
- 跨端数据校验采用 Zod
- Gemini 首版继续放在 Next 服务端接入
- 采用 Figma MCP 进行用户与 AI 协作开发

## 核心业务对象
围绕以下对象展开系统设计：
- user
- conversation
- message
- 可扩展对象：prompt template / tag / favorite / attachment metadata

默认核心主线为：
- 用户进入系统
- 创建或打开会话
- 发送消息并获得 AI 回复
- 历史消息写入数据库
- 用户可查看、管理和恢复会话

## Phase 路线

### Phase 1：聊天主链路 MVP
目标：先做出一个可用、好看、结构正确的聊天网页。

包括：
- 聊天页面
- 消息列表
- 输入框与发送按钮
- 加载状态与错误反馈
- Gemini 非流式调用
- 单页内多轮对话

完成标志：
- 用户能发送消息并收到 Gemini 回复
- 页面可以连续对话
- API Key 不暴露给客户端

### Phase 2：体验优化与结构整理
目标：在不引入重度返工的前提下，优化聊天体验并整理前端结构，为数据库接入做准备。

包括：
- 输入体验优化
- 自动滚动优化
- 更清晰的消息状态反馈
- 组件边界整理
- 前后端数据结构进一步统一

完成标志：
- 聊天交互更顺手
- 代码结构更适合承接持久化和会话系统

### Phase 3：Supabase 持久化与会话 CRUD 系统
目标：完成课程设计所需的数据库主线，让会话与消息真正进入数据库。

包括：
- 接入 Supabase
- 建立 users / conversations / messages 等核心数据模型
- 新建会话
- 查看会话列表
- 恢复历史会话
- 重命名会话
- 删除会话
- 消息持久化与读取

完成标志：
- 会话数据可持久化
- 用户界面体现完整的会话管理能力
- 系统具备明确的数据库支撑价值

### Phase 4：产品能力增强与生成体验升级
目标：在数据库主线稳定后，把当前系统从“可用的数据库驱动聊天产品”推进为“具备会话级控制、生成过程可感知、消息与会话更可管理，并能逐步接入联网与多模态能力”的完整 AI 产品原型。

本阶段默认同时服务三类目标：
- 产品目标：让聊天更自然、更顺手、更像真正可长期使用的 AI 产品
- 工程目标：在继续扩展能力的同时，保持前后端边界清晰、状态流稳定、架构可持续演进
- 学习目标：让新增能力尽量建立在清晰、可解释、可复盘的技术方案上，而不是短期拼接

本阶段的设计原则：
- 继续优先服务用户感知，而不是按底层模块机械拆功能
- 每一批能力都需要对应的技术落点，避免只做 UI 表层包装
- 会话仍是核心组织单位，优先采用“会话级独立配置、会话级独立状态”的产品思路
- 全局偏好可以存在，但主要用于新建会话时提供默认值，而不是替代会话本身的独立配置
- 视觉方向继续保持克制、浅色、利于专注，不因能力变多而堆叠过度操作区

本阶段包含但不限于：
- 模型注册表与多 provider 扩展
- 统一 AI 兼容层
- 流式输出与增量渲染
- 中断生成与更细粒度的消息状态
- 会话级提示词与会话级能力开关
- 模型选择能力与会话级记忆
- 消息复制、编辑、分支等消息侧增强
- 会话归档、恢复、收藏等管理增强
- 联网与多模态输入支持
- 后续更多 AI 扩展能力的预留位

本阶段按子阶段推进，详见 [[plan/phase/phase_4|phase_4]]。

完成标志：
- 回复可以逐步显示，生成过程可感知、可中断、可恢复控制
- 会话级提示词、模型选择和相关能力开关形成稳定的控制入口
- 消息与会话管理能力从“能用”提升到“更完整、更像真实产品”
- 联网与多模态能力至少完成首轮可用接入
- 系统具备继续承接更多模型、更多能力和更多产品增强项的清晰演进方向

### Phase 5：AI 能力扩展与课程答辩打磨
目标：在主链路和数据库能力稳定后，补充可展示亮点并整理课程交付材料。

包括：
- 部署上线与公网可访问能力整理
- 基础安全性与上线侧配置整理
- 文档与数据库设计对齐
- 安装说明与演示路径整理
- 答辩展示优化

完成标志：
- 产品可演示
- 具备上线展示条件与基础安全边界
- 文档、代码、答辩三者叙事一致
- 项目具备完整课程设计交付能力

## 阶段位置
WebAI 已经具备 `Phase 3` 的数据库驱动聊天主链路，并进入 `Phase 4` 的产品能力增强阶段。

这里可以理解为：
- `Phase 3` 主线已经成立：
  - 会话 CRUD 可支撑页面演示
  - 消息持久化可支撑历史恢复
  - 页面主链路已经可以稳定演示
- `Phase 3.6` 可以开始验收与材料整理：
  - `RLS` 验证
  - 数据库说明补全
  - migration 与表设计映射整理
- `Phase 4` 作为产品增强阶段继续推进：
  - `system_prompt` 已打通
  - 模型注册表已落库
  - AI 层已拆为 `Gemini` 与 `OpenAI compatible`
  - 前端模型选择入口连接会话级模型配置
  - 当前应按已定义子阶段继续收口与扩展


## 设计边界与取舍
- 浏览器只调用 Next 提供的服务端接口
- Gemini API Key 仅存在于服务端
- 所有跨端数据都应通过统一 schema 校验
- 数据库中的核心对象优先围绕会话与消息展开
- 优先完成数据库课程设计所需的主线能力
- 不优先投入复杂用户系统
- 不提前做过多高阶 AI 扩展
- 产品体验与数据库价值要同时可见，但优先保证主链路可交付
