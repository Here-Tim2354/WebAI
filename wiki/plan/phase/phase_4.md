# Phase 4

## 目标
- 在 `Phase 3` 数据库主线稳定后，把当前系统继续推进为更完整的 AI 产品原型
- 让用户先明显感知到“更顺手、更自然、更可控”的聊天体验升级
- 让新增能力建立在清晰、稳定、可扩展的技术结构之上，而不是靠短期拼接
- 让本阶段的推进过程同时服务产品完善、课程展示和个人学习

## 本阶段定位
- `Phase 4` 不再是“是否进入多模型和增强能力”的讨论阶段，而是正式推进阶段
- 本阶段从以下基础继续推进：
  - 会话级 `system_prompt` 链路已打通
  - 模型注册表已收口为 `model_catalog + model_fetched`
  - AI 层围绕 Gemini 原生接口组织
  - 前端模型选择入口连接模型注册表
- 因此本阶段默认从“已有基础之上的增量增强”开始，而不是从零规划生成系统

## 核心原则

### 产品原则
- 优先做出真正好用、好看、顺手的产品体验，而不是为了尽快实现走捷径
- 每一批都应优先解决用户能明显感知到的体验问题
- 视觉继续保持浅色、克制、利于专注，不因功能增多而堆满界面

### 工程原则
- 每个能力都要有对应的数据边界、状态边界和服务端边界
- 新能力应尽量基于现有注册表、会话对象和消息对象继续演进
- 能逐步落库的对象优先逐步落库，再与前端一同收口
- 不为短期方便牺牲长期可维护性

### 学习原则
- 当前项目采用的技术栈对用户仍有较强学习价值
- 因此本阶段默认偏好“结构清楚、便于理解和复盘”的实现路线
- 若某方案虽然能更快上线，但会显著削弱代码可读性、状态清晰度或学习价值，则不作为默认优先方案

## 会话级思路
- `Phase 4` 默认进一步强化“会话级独立”的产品思路
- 重点包括：
  - 当前会话独立的 `system_prompt`
  - 当前会话独立的模型选择
  - 当前会话独立的联网等能力开关
  - 当前会话独立记忆其偏好与控制项
- 全局偏好可以存在，但主要用于：
  - 用户弹出窗口中的全局设置
  - 新建会话时提供默认值
  - 为用户提供稳定的默认偏好入口
- 全局偏好不应覆盖会话本身的独立配置

## 关键产品语义

### 会话级提示词
- 当前头部已有图标入口
- 该入口在 `Phase 4` 中应正式成为“当前会话 AI 控制入口”的第一版
- 当前版本以 `system_prompt` 为核心
- 后续可逐步承接更多会话级控制项，但不应一次性做成庞杂设置中心

### 消息编辑
- 当前定义为覆盖式编辑
- 默认不把复杂版本浏览 UI 作为当前阶段前置条件
- 若后续需要保留审计或历史能力，应优先从数据层设计入手，而不是先做复杂可视化

### 消息分支
- 当前定义不是在单个会话内部绘制复杂树状结构
- 当前定义为：把现有上下文迁移为一个新会话
- 新会话默认沿用旧会话的：
  - 模型选择
  - 会话级提示词
  - 相关会话级能力开关
  - 其余必要的上下文设置

### 联网能力
- 第一原则是显式控制，而不是默认交给模型自行决定
- 默认以会话级常驻开关的方式组织
- 每个会话应能记住自己的联网选择
- 全局设置可决定新会话的默认开关状态，但不替代会话本身的选择

### 思考档位
- 当前按 Gemini 3 Flash 的 thinking 规格处理为“档位”，而不是“开关”
- 会话级保存 `thinking_level`，取值为 `minimal / low / medium / high`
- Gemini 3 Flash 不能完全关闭 thinking，最低档是 `minimal`
- assistant 的思考内容展示为 thought summary，默认折叠，不混入最终回答正文

### 多模态输入
- 当前方向只包括图片和文件输入
- 用户既可以通过按钮上传，也可以通过复制粘贴触发输入
- 系统应尽量自动识别输入类型
- 视频与音频明确不作为当前产品方向

## 分阶段安排

### 4.1 生成体验与会话控制基础升级
- 目标：
  - 先把“发出一条消息后，用户实际感受到的整个生成过程”做顺、做稳、做美观
- 重点：
  - 流式输出
  - assistant 回复的增量渲染
  - 中断生成
  - 更细粒度的消息状态管理
  - Gemini 模型选择入口收口
  - 当前会话模型选择记忆
  - 会话级 `system_prompt` 的正式可编辑入口
  - 当前会话控制项的第一轮组织方式
- 产出：
  - 用户能明显感知到更自然、更快反馈、更可控的生成体验
  - 会话头部控制区具备清晰而克制的交互逻辑

#### 4.1.1 能力范围
- 生成链路包含：
  - 纯流式聊天接口
  - assistant 回复增量渲染
  - 显式中断生成
  - 更细粒度的消息状态
  - 会话级 `system_prompt` 的正式编辑入口
  - 当前会话模型选择记忆
  - 控制区的第一轮统一组织方式
  - 会话级联网开关的后端与数据库主链路
  - `Gemini URL Context` 的请求级后端接入
  - 会话级联网前端真实开关
  - 会话级思考档位的数据库字段、前端入口和请求传参
  - Gemini thought summary 进入 assistant 消息 metadata，并在消息气泡中默认折叠展示
  - `Gemini URL Context` 的前端常驻输入入口与请求传参
- 产品语义：
  - 首页草稿态下的模型和提示词只在本页暂存，发送首条消息后才落库
  - 当前会话一旦存在，其模型选择、`system_prompt` 与联网配置即视为会话级配置
  - 思考档位同样视为会话级配置；空白首页先暂存在草稿态，首条消息创建会话时一并落库
  - 联网能力当前已形成真实会话级开关；空白首页草稿态下切换的值会在首条消息创建会话时一并落库
  - `URL Context` 当前已形成请求级前端入口：支持输入、确认、删除、上限提示、发送时传参，并在发送后清空与收起
  - assistant 中断后数据库状态应写为 `cancelled`，取消请求应绑定到当前 assistant 消息，避免影响同一会话后续回复
- 系统边界：
  - `messages.status` 保存消息状态
  - `conversations.model_id` 保存用户启用后的 `model_fetched.id`
  - 会话级联网配置默认开启
  - 会话级思考档位默认值为 `minimal`
  - Gemini 调用层统一按 delta 流消费
  - Gemini provider 已能区分普通 text delta 与 `part.thought` thought summary delta
  - 聊天接口使用 NDJSON 事件流
  - `/api/chat` 已支持可选 `urls`
  - `Gemini` 调用层已能按请求条件注入联网工具与 `urlContext`
  - 前端展示层在消息渲染上做了本地 reveal 与 markdown 实时渲染

#### 4.1.1 继续打磨
- 仍建议继续补的内容包括：
  - 流式展示节奏与过渡动画的细修
  - `reduced motion` 等边界场景的最小回归验证
  - 控制区与提示词弹窗的残余视觉细节收口
  - 让阶段文档继续贴近用户能感知到的系统行为

#### 4.1.2 下一步重点
- 在 `4.1.1` 主链路成立后，下一步默认优先：
  - 继续提升生成过程的稳定感与自然感
  - 把状态反馈文案、移动端与桌面端体验继续收口
  - 继续收口联网开关与 `URL Context` 的排版、反馈动画与能力提示文案
  - 继续验证 `URL Context` 在桌面端与移动端下的可读性、输入节奏和边界反馈
  - 为后续 `4.2` 的消息复制 / 编辑 / 分支能力保持边界清晰

### 4.2 消息侧增强与会话分支能力
- 目标：
  - 让消息从“仅可阅读”升级为“可操作、可调整、可继续推演”的工作对象
- 重点：
  - 消息复制
  - 消息编辑
  - 从消息发起分支到新会话
  - 分支后沿用原会话的关键设置
  - 围绕消息操作的反馈与状态提示
- 产出：
  - 用户可以围绕现有消息继续修改、延展和重走上下文
  - 会话之间的演化关系开始形成更清楚的产品语义

#### 4.2 能力范围
- 消息操作链路包含：
  - 消息复制入口
  - user 消息覆盖式编辑
  - 编辑后由服务端直接流式重新生成 assistant 回复
  - 最新 assistant 消息重新生成入口
  - 从 assistant 消息创建新会话分支
  - 分支会话沿用原会话模型、提示词与联网开关
  - 分支创建过程的顶部工作区通知
  - `URL Context` 已开始写入 user 消息级 metadata，为重新生成与后续多模态输入保留上下文
  - user 消息已能轻量展示已附加的 URL Context 数量和链接入口
  - user 消息编辑态已支持修改 URL Context，并与正文共用保存按钮进入重新生成流
  - 聊天输入区、消息操作区、顶部模型选择和侧栏已进一步统一到小圆角视觉语言
  - 主要滚动区域使用 `OverlayScrollbars`，悬停提示使用项目内轻量 Tooltip
  - 消息流式输出期间的滚动策略已补强：用户 wheel / touch 上移时会暂停自动吸底，并通过圆形向下箭头提供回到底部入口
- 产品语义：
  - 复制优先使用 `Clipboard API`，失败后继续尝试隐藏 `textarea + execCommand` 降级通道
  - 编辑消息不再由前端拆成两个请求，而是由消息 PATCH 接口完成“编辑、截断、重新生成”一条流式链路
  - 编辑消息时如果只修改 URL Context，仍会保存到该 user 消息 metadata 并重新生成后续 assistant 回复
  - assistant 重新生成只允许作用于当前会话最后一条 assistant 消息，并沿用当前会话模型、提示词、联网开关与对应 user 消息保存的 URL Context
  - 如果重新生成前输入了新的 URL Context，重新生成会使用新 URL 并覆盖对应 user 消息的 metadata
  - URL Context 的消息级展示只作为辅助信息，不改变当前输入区的发送与清空行为
  - 编辑消息与删除后续消息依赖数据库 RPC 保证原子性，避免半更新上下文
  - 分支不是会话内树状结构，而是复制目标消息之前的上下文到新会话
  - 分支复制不再依赖 `created_at <= target.created_at` 判断边界，而是按完整消息列表定位目标消息后截断，并保留消息 metadata
  - 输入框文本草稿保留在 `ChatInput` 本地，发送时再把内容上交工作区，减少长文本输入时的全局重渲染
  - textarea 保留原生滚动条能力，避免 OverlayScrollbars 直接接管输入元素导致滚轮失效
  - `ScrollArea` 只通过官方 React wrapper 初始化 OverlayScrollbars，避免外部库改写 React 正在管理的 DOM 子树
  - `ScrollArea` 对外 ref 需要指向 OverlayScrollbars 的真实 viewport；初始化延迟时会通过 `initialized` 事件重新发布 viewport ref
- 继续观察：
  - 消息编辑 RPC、相关 policy 与消息级 metadata migration 需要和数据库环境保持一致；后续如果继续扩表，迁移也要同步跟进
  - 已创建过的旧分支会话如果存在相同 `created_at` 导致的乱序，需要单独 SQL 修复，不应由应用代码隐式修改历史数据
  - 旧独立 `/api/chat/regenerate` 入口已删除；当前重新生成入口为 assistant 消息专用的 `/api/messages/[messageId]/regenerate`
  - 当前 Tooltip 为无 Portal 的 CSS-only 方案，稳定性优先；如果后续出现裁切，再补专门的安全浮层策略
- 阶段位置：
  - `Phase 4.2` 的核心消息操作链路可以作为后续阶段的基础
  - 后续保留运行观察和旧分支历史数据修复项
  - 主线继续推进 `Phase 4.3` 与 `Phase 4.4`

### 4.3 会话管理增强与组织能力扩展
- 目标：
  - 让会话从数据库 CRUD 对象继续升级为更完整的产品工作对象
- 重点：
  - 会话归档
  - 归档恢复
  - 归档区管理
  - 收藏能力
  - 相关扩展对象的逐步落库
- 产出：
  - 会话管理能力从“可演示”提升到“更接近真实可持续使用”
  - 数据库侧的扩展对象开始形成更丰富的课程展示价值

#### 4.3 能力范围
- 会话组织能力包括：
  - 会话收藏 / 取消收藏
  - 会话归档 / 恢复
  - 归档区入口
  - 收藏区入口与收藏会话跳转
  - 用户头像二级菜单承接个人账户、收藏、归档区、Gemini 设置与退出登录
  - 用户头像二级菜单承接更新日志入口
  - 新建会话采用乐观更新：浏览器先生成会话 UUID 并插入侧栏，服务端使用同一 UUID 落库，失败时回滚列表、消息缓存与激活会话
  - 收藏区与归档区在登录后尝试后台预取，主动打开时再刷新一次
- 数据库对象包括：
  - `20260427083000_phase4_conversation_organization.sql`
  - 包含 `favorites`、`conversations.archived_at`、RLS 与索引
- 继续观察：
  - Supabase 环境需要包含收藏、归档相关 migration
  - 收藏区、头像菜单入口和归档区需要在真实多会话数据下保持清楚
  - 新建会话乐观更新需要继续观察弱网、重复点击和首条消息紧接发送时的状态一致性
  - 移动端与多数据场景仍值得继续观察

#### 4.3 账户与版本提示补充
- 个人账户能力包括：
  - 修改昵称
  - 上传头像
  - 修改密码
  - 注销账户
- 注销账户由服务端 admin client 删除当前 Auth 用户，并清理当前用户在 `profile_avatars` 与 `message_attachments` 中的对象；数据库侧继续依赖用户外键级联清理 profile、conversation、message 与 favorite 等核心记录
- 更新日志能力包括：
  - 当前版本日志从 `wiki/Optimization/version/V1.1.md` 读取
  - 左下角头像菜单提供常驻入口
  - 登录后自动提示当前版本内容
  - 同一浏览器每天最多自动提示一次
  - 用户可选择不再弹出本次更新内容

### 4.4 联网与多模态输入扩展
- 目标：
  - 在主聊天体验和管理能力稳定后，逐步把系统扩展到更真实的 AI 使用场景
- 重点：
  - 会话级联网开关
  - 图片输入的首轮接入
  - 文件输入的首轮接入
  - 模型能力与输入能力的注册表映射继续完善
- 产出：
  - 当前系统开始具备更真实的外部信息获取与多模态输入能力
  - 会话级控制和模型能力边界保持一致

#### 4.4 能力范围
- 文件与图片输入链路包括：
  - `messages.metadata` 在保留 `urls` 的同时新增 `attachments`
  - 新增私有 Storage bucket `message_attachments`
  - 模型能力字段从 `supports_file_search` 调整为 `supports_files`
  - 输入区支持上传、粘贴、拖拽图片和文件
  - URL、图片、文件统一进入“修改附加项”窗口
  - Gemini 设置支持按用户 API Key / URL 拉取模型，并在用户私有模型列表中启用 / 停用
  - user 消息支持图片缩略图、点击放大和文件条展示
  - 发送、编辑、重新生成、分支均沿用消息级 metadata 上下文
  - Gemini 输入组装已支持图片、PDF、文本类文件与 Excel 转换后的 CSV
  - Storage object key 已改为安全路径，原始文件名只保留在 metadata 中展示，避免中文/空格文件名触发 Storage `Invalid key`
  - Word / PPT 不作为上传支持类型，文件选择阶段直接提示支持范围
  - Markdown 正文渲染支持基础 LaTeX 公式和 `\ce{}` 化学式语法
  - 编辑带附件消息优先走数据库 RPC，并保留直接 update + delete 兜底路径，降低 RPC 单点失败风险
- 阶段判断：
  - 多模态附件链路已经进入可体验阶段，但产品体验还需要继续打磨
  - 文件选择、上传反馈、弹窗保存、编辑带附件消息、重新生成、分支、历史恢复、LaTeX / 化学式渲染和移动端图片预览都需要继续用真实数据深度回归
- 产品边界：
  - 视频与音频明确不作为当前产品方向
  - 正文和附加项只要存在其一即可发送
  - 删除附加项后会尝试清理不再被任何消息引用的 Storage 对象
- 继续打磨：
  - “修改附加项”窗口的添加、删除、保存、取消、错误提示和禁用状态
  - 真实附件上传后的 UI 反馈和失败恢复
  - 编辑带附件 user 消息时的数据库原子更新 / fallback 与前端状态同步
  - assistant 重新生成和分支对附件 metadata 的继承稳定性
  - LaTeX / 化学式渲染在流式输出、中文段落和代码块附近的边界表现
  - TikZ / chemfig 不进入聊天渲染链路，避免引入 LaTeX 编译级复杂度
  - 移动端图片缩略图、页面级预览和私有 Storage 代理加载体验
  - 自动清理失败时是否需要后台补偿任务

## 不作为前置条件的内容
- 不要求一次性做完所有 provider
- 不要求在首批就把所有文件类型处理都做到完整视觉理解
- 不要求在当前阶段前半程就进入复杂多 Agent 体系
- 不要求为了更快上线而提前牺牲产品一致性和代码整洁度

## 完成标志
- 生成过程对用户而言更快、更自然、更可控
- 会话级提示词、模型选择和能力开关形成稳定控制逻辑
- 消息和会话都具备更强的管理与操作能力
- 联网与多模态输入至少完成首轮可用接入
- 阶段文档可以继续承接后续 `4.5+` 的新能力，而无需推翻前面阶段结构

## 一句话总结
- `Phase 4` 的核心不是“继续往聊天系统里加功能”，而是在守住产品体验、代码整洁与架构演进质量的前提下，把 `WebAI` 逐步推进为一个更完整、更可控、更值得长期打磨的 AI 产品原型。
