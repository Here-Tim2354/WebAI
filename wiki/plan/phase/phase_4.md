# Phase 4

## 目标
- 在 `Phase 3` 数据库主线稳定后，把当前系统继续推进为更完整的 AI 产品原型
- 让用户先明显感知到“更顺手、更自然、更可控”的聊天体验升级
- 让新增能力建立在清晰、稳定、可扩展的技术结构之上，而不是靠短期拼接
- 让本阶段的推进过程同时服务产品完善、课程展示和个人学习

## 本阶段定位
- `Phase 4` 不再是“是否进入多模型和增强能力”的讨论阶段，而是正式推进阶段
- 当前已经成立的基础包括：
  - 会话级 `system_prompt` 链路已打通
  - 模型注册表已落为 `ai_models + provider 子表`
  - AI 层已拆为 `Gemini` 与 `OpenAI compatible`
  - 前端首轮模型选择入口已接入
- 因此本阶段默认从“已有基础之上的增量增强”开始，而不是从零规划多模型系统

## 核心原则

### 产品原则
- 优先做出真正好用、好看、顺手的产品体验，而不是为了尽快实现走捷径
- 每一批都应优先解决用户能明显感知到的体验问题
- 视觉继续保持浅色、克制、利于专注，不因功能增多而堆满界面

### 工程原则
- 每个能力都要有对应的数据边界、状态边界和服务端边界
- 新能力应尽量基于现有注册表、会话对象和消息对象继续演进
- 能逐步落库的对象优先逐步落库，再与前端一同收口
- 不为短期方便牺牲长期可维护性

### 学习原则
- 当前项目采用的技术栈对用户仍有较强学习价值
- 因此本阶段默认偏好“结构清楚、便于理解和复盘”的实现路线
- 若某方案虽然能更快上线，但会显著削弱代码可读性、状态清晰度或学习价值，则不作为默认优先方案

## 会话级思路
- `Phase 4` 默认进一步强化“会话级独立”的产品思路
- 重点包括：
  - 当前会话独立的 `system_prompt`
  - 当前会话独立的模型选择
  - 当前会话独立的联网/搜索等能力开关
  - 当前会话独立记忆其偏好与控制项
- 全局偏好可以存在，但主要用于：
  - 用户弹出窗口中的全局设置
  - 新建会话时提供默认值
  - 为用户提供稳定的默认偏好入口
- 全局偏好不应覆盖会话本身的独立配置

## 关键产品语义

### 会话级提示词
- 当前头部已有图标入口
- 该入口在 `Phase 4` 中应正式成为“当前会话 AI 控制入口”的第一版
- 当前版本以 `system_prompt` 为核心
- 后续可逐步承接更多会话级控制项，但不应一次性做成庞杂设置中心

### 消息编辑
- 当前定义为覆盖式编辑
- 默认不把复杂版本浏览 UI 作为当前阶段前置条件
- 若后续需要保留审计或历史能力，应优先从数据层设计入手，而不是先做复杂可视化

### 消息分支
- 当前定义不是在单个会话内部绘制复杂树状结构
- 当前定义为：把现有上下文迁移为一个新会话
- 新会话默认沿用旧会话的：
  - 模型选择
  - 会话级提示词
  - 相关会话级能力开关
  - 其余必要的上下文设置

### 联网与搜索
- 第一原则是显式控制，而不是默认交给模型自行决定
- 默认以会话级常驻开关的方式组织
- 每个会话应能记住自己的联网/搜索选择
- 全局设置可决定新会话的默认开关状态，但不替代会话本身的选择

### 多模态输入
- 长期方向包括图片、文件、视频输入
- 用户既可以通过按钮上传，也可以通过复制粘贴触发输入
- 系统应尽量自动识别输入类型
- 但首批实现可优先收敛到图片输入，再逐步扩展到文件和视频

## 分阶段安排

### 4.1 生成体验与会话控制基础升级
- 目标：
  - 先把“发出一条消息后，用户实际感受到的整个生成过程”做顺、做稳、做美观
- 重点：
  - 流式输出
  - assistant 回复的增量渲染
  - 中断生成
  - 更细粒度的消息状态管理
  - 模型选择入口收口
  - 当前会话模型选择记忆
  - 会话级 `system_prompt` 的正式可编辑入口
  - 当前会话控制项的第一轮组织方式
- 产出：
  - 用户能明显感知到更自然、更快反馈、更可控的生成体验
  - 会话头部控制区具备清晰而克制的交互逻辑

#### 4.1.1 当前完成情况
- 当前已经完成的主链路包括：
  - 纯流式聊天接口
  - assistant 回复增量渲染
  - 显式中断生成
  - 更细粒度的消息状态
  - 会话级 `system_prompt` 的正式编辑入口
  - 当前会话模型选择记忆
  - 控制区的第一轮统一组织方式
- 当前实现语义：
  - 首页草稿态下的模型和提示词只在本页暂存，发送首条消息后才落库
  - 当前会话一旦存在，其模型选择与 `system_prompt` 即视为会话级配置
  - 联网入口当前统一展示，但仍处于置灰占位状态
  - assistant 中断后数据库状态应写为 `cancelled`
- 当前技术落点：
  - `messages.status` 已落到数据库
  - `conversations.model_id` 已落到数据库
  - AI provider 层统一按 delta 流消费
  - 聊天接口使用 NDJSON 事件流
  - 前端展示层在消息渲染上做了本地 reveal 与 markdown 实时渲染

#### 4.1.1 当前剩余事项
- 当前仍建议继续补的内容包括：
  - 流式展示节奏与过渡动画的细修
  - `reduced motion` 等边界场景的最小回归验证
  - 控制区与提示词弹窗的残余视觉细节收口
  - 阶段文档与实现真相的持续同步

#### 4.1.2 下一步重点
- 在 `4.1.1` 主链路成立后，下一步默认优先：
  - 继续提升生成过程的稳定感与自然感
  - 把状态反馈文案、移动端与桌面端体验继续收口
  - 明确联网/搜索开关的真实数据契约与接入方式
  - 为后续 `4.2` 的消息复制 / 编辑 / 分支能力保持边界清晰

### 4.2 消息侧增强与会话分支能力
- 目标：
  - 让消息从“仅可阅读”升级为“可操作、可调整、可继续推演”的工作对象
- 重点：
  - 消息复制
  - 消息编辑
  - 从消息发起分支到新会话
  - 分支后沿用原会话的关键设置
  - 围绕消息操作的反馈与状态提示
- 产出：
  - 用户可以围绕现有消息继续修改、延展和重走上下文
  - 会话之间的演化关系开始形成更清楚的产品语义

### 4.3 会话管理增强与组织能力扩展
- 目标：
  - 让会话从数据库 CRUD 对象继续升级为更完整的产品工作对象
- 重点：
  - 会话归档
  - 归档恢复
  - 归档区管理
  - 收藏能力
  - 会话搜索
  - 相关扩展对象的逐步落库
- 产出：
  - 会话管理能力从“可演示”提升到“更接近真实可持续使用”
  - 数据库侧的扩展对象开始形成更丰富的课程展示价值

### 4.4 联网、搜索与多模态输入扩展
- 目标：
  - 在主聊天体验和管理能力稳定后，逐步把系统扩展到更真实的 AI 使用场景
- 重点：
  - 会话级联网开关
  - 显式搜索能力接入
  - 搜索结果与消息生成链路的协同
  - 图片输入的首轮接入
  - 为文件、视频输入预留结构
  - 模型能力与输入能力的注册表映射继续完善
- 产出：
  - 当前系统开始具备更真实的外部信息获取与多模态输入能力
  - 会话级控制和模型能力边界保持一致

### 4.5+ 预留扩展阶段
- 说明：
  - `Phase 4` 不应被视为封闭阶段，后续仍会继续增加新的增强功能
  - 因此当前阶段文档需要显式保留后续子阶段扩展位
- 后续可能承接但暂不写死的方向包括：
  - 更多 provider 接入
  - 更强的工具使用能力
  - 更完整的搜索体系
  - 文件与视频输入的正式落地
  - 更深入的 Agent / reasoning 能力
  - 更多产品化增强项

## 当前不作为前置条件的内容
- 不要求一次性做完所有 provider
- 不要求在首批就把所有多模态输入类型全部落地
- 不要求在当前阶段前半程就进入复杂多 Agent 体系
- 不要求为了更快上线而提前牺牲产品一致性和代码整洁度

## 完成标志
- 生成过程对用户而言更快、更自然、更可控
- 会话级提示词、模型选择和能力开关形成稳定控制逻辑
- 消息和会话都具备更强的管理与操作能力
- 联网与多模态输入至少完成首轮可用接入
- 阶段文档可以继续承接后续 `4.5+` 的新能力，而无需推翻前面阶段结构

## 一句话总结
- `Phase 4` 的核心不是“继续往聊天系统里加功能”，而是在守住产品体验、代码整洁与架构演进质量的前提下，把 `WebAI` 逐步推进为一个更完整、更可控、更值得长期打磨的 AI 产品原型。
