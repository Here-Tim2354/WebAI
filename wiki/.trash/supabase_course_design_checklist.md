# Supabase 课程设计注意事项与执行步骤

## 文档目的

本文用于指导 `WebAI` 项目在使用 Supabase 进行数据库设计与实现时，尽量同时满足：

- 产品主线需求
- `Phase 3` 的持久化目标
- 华南农业大学数据库课程设计的硬性要求

它不是课程报告正文，而是一份面向实现与答辩准备的执行清单。

---

## 一、先明确：课程真正考察的不是 Supabase 本身

使用 Supabase 没问题，但课程老师重点看的不是“是否用了云数据库平台”，而是：

- 是否真的使用数据库存储核心数据
- 用户界面上是否体现了完整的增删查改
- 数据库设计是否从需求推导而来
- 是否能解释实体、联系、主键、外键、索引、约束
- 是否能把需求分析、数据库设计、系统实现串成闭环

所以在项目表达上，应把 Supabase 定位为：

- `Postgres 数据库 + Auth + Storage + 平台化能力`

而不是把它当成答辩主角。

更合适的叙述方式是：

- 本系统基于 Supabase 提供的 PostgreSQL 数据库完成核心数据持久化
- 使用 Supabase 的数据库能力实现会话与消息管理
- 根据需要使用 Auth / RLS / Storage 作为权限与扩展支持

---

## 二、设计时必须特别注意的点

### 1. 数据字典必须从用户视角写，不是表结构

课程要求里的“数据字典”强调的是用户在界面上看到、输入、查询的数据，不是数据库字段列表。

因此不能直接把下面这些当作数据字典：

- `conversation.id`
- `messages.role`
- `profiles.created_at`

更合理的写法应该是从界面和用例出发，例如：

- 会话标题
- 会话创建时间
- 最近更新时间
- 用户输入的问题内容
- AI 回复内容
- 消息发送时间
- 会话查询关键词
- 会话重命名输入值

正确顺序应当是：

1. 先梳理用户在界面上会看见和操作哪些数据
2. 再抽象出实体与联系
3. 最后形成数据库表结构

### 2. CRUD 必须在界面上真实可演示

课程要求明确规定用户界面必须支持增删查改。

对当前 WebAI 项目，建议至少做到：

- 增：新建会话、发送消息并写入数据库
- 查：查看会话列表、查看历史消息、恢复指定会话
- 改：重命名会话
- 删：删除会话

如果时间允许，还可以补充：

- 改：编辑会话备注、设置标签
- 删：删除单条消息

注意：

- 不能只有“聊天发送成功”就算完成数据库课程设计
- 必须让老师在界面上看到数据库主线对应的操作闭环

### 3. 核心实体要清楚、可解释

当前项目比较稳妥的核心实体建议是：

- 用户 `user / profile`
- 会话 `conversation`
- 消息 `message`

基础关系建议：

- 一个用户可以拥有多个会话
- 一个会话可以拥有多条消息
- 一条消息只属于一个会话

如果想增强数据库设计展示度，可扩展：

- 标签 `tag`
- 会话标签关联 `conversation_tag`
- 收藏 `favorite`
- 附件元数据 `attachment`

但不建议为了显得复杂而过度堆表。课程答辩更看重“每个实体是否来自需求”。

### 4. 表设计尽量规范化，不要把业务全塞进 JSON

Supabase 底层是 Postgres，非常适合做规范化的关系设计。

应优先把稳定的核心数据拆到结构化字段里，例如：

- 会话标题
- 用户 ID
- 消息角色
- 消息内容
- 创建时间
- 更新时间

不要一开始就把大量业务信息放到一个 JSON 字段里，否则会带来这些问题：

- 不利于说明实体关系
- 不利于课程文档中的逻辑结构设计
- 不利于索引设计和查询解释
- 不利于答辩时说明主键、外键和约束

JSON 字段只适合承载：

- 可变的扩展元数据
- 不稳定、暂时不适合拆表的附加信息

### 5. 权限设计是加分项，但不能喧宾夺主

Supabase 的 RLS 很适合做课程里的“用户权限控制”展示。

建议在后续设计中体现：

- 普通用户只能读取自己的会话
- 普通用户只能读取自己的消息
- 普通用户只能修改自己的会话
- 普通用户不能访问他人的数据

但在表达顺序上要注意：

1. 先讲需求和数据模型
2. 再讲关系模式和约束
3. 最后讲 RLS 和权限策略

也就是说：

- RLS 是数据库安全设计的亮点
- 不是替代 ER 设计和关系模式设计的主线

### 6. 索引、约束、可空性要提前考虑

课程要求里明确提到了：

- 哪些字段需要建立索引
- 是否需要触发器
- 是否需要存储过程或函数
- 是否要考虑空值率和扩展性

对当前项目，建议优先解释这些点：

- `conversations.user_id` 需要索引，因为常按用户查会话
- `conversations.updated_at` 需要索引，因为会按更新时间排序
- `messages.conversation_id` 需要索引，因为会按会话查消息
- `messages.created_at` 可配合排序或时间范围查询

约束层面建议说明：

- 主键不能为空且唯一
- 外键保证消息必须属于某个会话
- 消息角色值应限制在合法范围内
- 标题长度应有合理限制

### 7. 设计要留扩展余地，但不要提前过度实现

课程设计里需要说明数据库如何适应需求变化。

对本项目，可以这样回答：

- 后续可扩展消息状态字段
- 后续可扩展附件表
- 后续可扩展会话标签和收藏能力
- 后续可扩展多模型记录、Token 统计、生成参数

但当前阶段实现上，仍应优先守住课程主线：

- 会话持久化
- 消息持久化
- 会话 CRUD

## 三、基于本项目的推荐执行步骤

下面这套步骤尽量同时服务：

- 项目开发推进
- 课程文档编写
- 后续答辩展示

### 第 1 步：先整理用户角色、界面和用例

先不要急着建表。

应先明确：

- 谁在使用系统
- 用户在界面上能做什么
- 每个操作会看到什么数据、输入什么数据

对当前项目，最小角色可以先只有：

- 普通用户

最小核心用例建议包括：

- 进入聊天页面
- 新建会话
- 发送消息
- 查看历史会话
- 打开某个历史会话
- 重命名会话
- 删除会话

这一步的产出应包括：

- 用户角色说明
- 功能结构
- 系统流程
- 用户视角的数据字典

### 第 2 步：从用例反推实体与联系

完成用例后，再抽实体。

对当前项目，推荐先固定为：

- 用户
- 会话
- 消息

关系建议：

- 用户与会话：一对多
- 会话与消息：一对多

如果后续要增强课程展示度，可再补：

- 会话与标签：多对多

这一步的产出应包括：

- 实体说明
- 联系说明
- ER 图草案

### 第 3 步：完成关系模式设计

在 ER 图之后，再落关系模式。

建议至少明确：

- 每张表的中文名与英文名
- 字段名
- 数据类型
- 是否允许为空
- 主键
- 外键
- 默认值
- 描述说明

对当前项目，第一版至少应覆盖：

- `profiles`
- `conversations`
- `messages`

如果不单独做 `profiles`，也要在文档里说清楚：

- 用户基础身份由 Supabase Auth 提供
- 业务核心数据通过用户 ID 关联到会话和消息

### 第 4 步：设计约束、索引和删除策略

这一步不要省略，因为它很像“数据库课程设计”。

需要明确：

- 会话删除后，消息是否级联删除
- 是否允许空标题
- 消息内容是否允许为空
- 是否需要限制消息角色枚举
- 哪些字段建立索引

建议优先写清：

- 主键约束
- 外键约束
- 唯一性约束
- 检查约束
- 级联策略
- 索引设计

### 第 5 步：再决定是否引入 Auth、RLS、Storage

这一步要看当前阶段目标，不要本末倒置。

推荐优先级：

1. 先完成核心表与 CRUD
2. 再补最基础的用户隔离
3. 最后按需要扩展 Auth / Storage

比较稳的策略是：

- 用 Supabase Auth 提供用户身份
- 用 RLS 控制“用户只能访问自己的会话和消息”
- Storage 暂不作为首要目标，除非你准备做附件功能

### 第 6 步：把数据库操作映射到页面功能

数据库设计完成后，要马上对应到页面，不然课程里会显得“数据库和应用脱节”。

建议对应关系写成这样：

- 新建会话 -> 插入 `conversations`
- 发送消息 -> 插入 `messages`
- 查看历史会话 -> 查询 `conversations`
- 打开历史会话 -> 查询某会话及其 `messages`
- 重命名会话 -> 更新 `conversations.title`
- 删除会话 -> 删除 `conversations` 及其关联消息

### 第 7 步：把 migration 作为课程材料的一部分

Supabase 的 migration 很适合拿来支撑“数据库模式演化”。

建议把后续 migration 文件视为：

- 数据库实现依据
- 模式设计结果
- 答辩时可展示的工程证据

这样做的好处是：

- 能说明数据库不是手工乱改的
- 能说明结构有演化过程
- 能和课程里的“数据库模式设计、实现过程”对应起来

## 四、当前项目最推荐的最小落地范围

为了兼顾课程要求、当前 phase 和开发成本，建议第一阶段只做最小但完整的数据库主线：

### 必做

- 会话表
- 消息表
- 会话列表查询
- 历史消息恢复
- 会话重命名
- 会话删除
- 聊天消息持久化

### 建议做

- 用户与会话的归属关系
- 基础 RLS
- 创建时间与更新时间维护
- 关键索引

### 可以后做

- 标签系统
- 收藏系统
- 附件上传
- 消息编辑
- 流式写入优化

## 五、答辩和文档时应避免的坑

### 1. 只讲 Supabase，不讲数据库设计

错误表达：

- 我们用了 Supabase，所以数据库部分已经完成

正确表达：

- 我们基于用户需求抽取了会话、消息等核心实体
- 在 Supabase PostgreSQL 中实现了关系模式、约束和索引
- 再通过前端界面完成会话与消息的增删查改

### 2. 只有聊天，没有完整 CRUD

如果只能发送消息、显示消息，而没有：

- 会话列表
- 会话重命名
- 会话删除

那课程主线会偏弱。

### 3. 数据字典直接抄数据库字段

这会直接偏离课程要求。

数据字典必须从“用户看到什么、输入什么、查询什么”来写。

### 4. 表结构无法解释来源

答辩时如果说不清：

- 为什么有这张表
- 为什么是一对多
- 为什么这个字段要建索引

就会影响数据库设计说服力。

### 5. 过度依赖平台默认能力，不总结自己的设计

Supabase 很方便，但课程成绩看的是你的小组设计能力。

所以必须在文档中主动补齐：

- ER 图
- 关系模式
- 数据字典
- 索引设计
- 权限设计
- 系统流程

---

## 六、现阶段建议的执行顺序

建议按下面顺序推进：

1. 先补需求分析中的角色、用例、数据字典
2. 明确 `用户 - 会话 - 消息` 的实体关系
3. 设计关系模式、主外键、约束、索引
4. 再在 Supabase 中落 migration
5. 接入前端，完成会话级 CRUD
6. 视时间补充 RLS 与权限控制
7. 最后整理 ER 图、表设计说明、系统实现说明

---

## 七、一句话结论

对这个项目来说，使用 Supabase 没问题，但必须始终记住：

- Supabase 只是数据库实现平台
- 课程考察的是“需求 -> 数据字典 -> ER 设计 -> 关系模式 -> CRUD 实现 -> 答辩表达”的完整闭环

只要你始终围绕这个闭环推进，Supabase 会是加分项；如果只把它当“Supabase 环境存数据工具”，反而容易在课程口径上失焦。
