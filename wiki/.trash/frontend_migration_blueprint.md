# Frontend Migration Blueprint

## 目标

本文件用于指导 WebAI 在 `Phase 3.5` 之后的一次前端架构迁移。

本轮迁移的核心不是单纯“换一套 UI”，而是同时完成以下几件事：

- 建立统一的设计语言
- 引入可持续维护的前端样式与组件底座
- 在不破坏聊天主链路的前提下重构页面骨架
- 让项目的视觉与交互更接近 `Vercel`、`ChatGPT`、`Perplexity` 代表的现代 AI 产品体验

本次迁移默认引入以下技术栈：

- `Tailwind CSS`
- `shadcn/ui`
- `Lucide React`
- `Motion`

同时将 `Untitled UI React` 作为重要的组件与页面模式参考源。

## 迁移定位

这是一次前端基建升级，不是一次数据库主线改题。

迁移完成后，项目应具备：

- 更稳定的样式系统
- 更清晰的组件层次
- 更一致的字体、颜色、图标和交互反馈
- 更适合继续扩展工作区、弹层、侧栏、空状态和消息界面的页面骨架

迁移期间必须守住：

- 聊天主链路不失效
- 会话管理不失效
- 登录链路不失效
- 历史消息恢复不失效
- 移动端基本可用

## 设计语言总纲

### 一句话定义

`Vercel` 式秩序感的浅色 AI 工作台，正文清爽克制，标题更锋利，蓝白为主，渐变只做空气感，交互体验更接近 `ChatGPT`，组件组织参考 `Untitled UI`。

### 视觉判断

- 整体气质：现代编辑感，偏西方科技网站常见的简约感
- 主视觉方向：浅色主页面，深色模式后续补上
- 页面结构：保留左侧栏，但升级为可折叠结构
- 内容主次：聊天主区始终是绝对主任务
- 动效原则：少而精，只保留真正改善感知的动效

### 已确认的设计决策

- 总体气质：`现代编辑感`
- 产品参考：`Vercel` 为 UI 原型参考；聊天体验参考 `ChatGPT` 与 `Perplexity`
- 主题策略：`浅色主页面，深色后续补上`
- 保留元素：`蓝白基调`、`合适的渐变底色`、`侧栏结构`
- 字体方向：正文以 `Geist Sans` 为主，代码与技术信息使用 `Geist Mono`，中文字体优先考虑 `MiSans`
- 标题方向：不追求花哨字体，优先通过更强的字重、字距、对比和版式建立识别度
- 信息密度：`偏疏朗`
- 圆角策略：`小圆角`
- 消息区方向：`更接近 ChatGPT`
- 侧栏策略：`可折叠`
- 登录页方向：维持蓝白渐变，但整体收得更高级

## 技术栈职责划分

### Tailwind CSS

定位：

- 作为项目的样式语法层和 token 承载层
- 负责布局、间距、响应式、主题变量和语义化设计 token

职责边界：

- 新组件优先使用 `Tailwind CSS` 表达布局和状态
- 不再继续扩大 `globals.css` 的手写样式范围
- 颜色、圆角、阴影、边框、间距、字体等统一映射到 design token

不做的事：

- 不把 `Tailwind` 变成“随手写原子类”的无序样式堆
- 不直接用大量裸色值破坏语义 token

参考：

- Tailwind CSS v4 官方安装与配置说明
- https://tailwindcss.com/blog/tailwindcss-v4

### shadcn/ui

定位：

- 作为基础交互组件底座
- 负责常见表单、弹层、菜单、提示、分隔、标签、骨架屏等基础件

职责边界：

- 以 `shadcn/ui` 作为组件源代码型底座
- 优先复用已有组件组合，而不是重复造基础轮子
- 组件样式优先通过语义 token 驱动，而不是局部硬编码覆盖

首批适用组件：

- `Button`
- `Input`
- `Textarea`
- `Dialog`
- `DropdownMenu`
- `Sheet`
- `Tooltip`
- `Badge`
- `Separator`
- `Skeleton`
- `Tabs`
- `Alert`

不做的事：

- 不把 `shadcn/ui` 当作视觉设计语言本身
- 不把业务复杂组件全部直接等同于基础组件库

参考：

- shadcn/ui Next.js 安装文档
- https://ui.shadcn.com/docs/installation/next

### Lucide React

定位：

- 作为全项目统一图标系统

职责边界：

- 导航、按钮、菜单、状态、工具条统一使用 `Lucide React`
- 图标只承担识别和辅助说明作用
- 保持线性、轻量、统一的描边风格

不做的事：

- 不为了“有图标”而堆积图标
- 不在每个文本块旁机械加图标

使用原则：

- 图标默认跟随文本颜色
- 控制尺寸统一，避免每个地方自己写一套尺寸逻辑
- 优先搭配文本出现，减少纯图标孤立动作

参考：

- Lucide 官方文档首页
- https://lucide.dev/

### Motion

定位：

- 作为页面级和结构级动效层

职责边界：

- 负责进入、退出、折叠、弹层、布局变化、消息插入反馈等动效
- 强调层级感、反馈感和切换感
- 默认只在真正影响体验的地方使用

优先使用场景：

- 登录页和空态的轻量入场
- 侧栏折叠和展开
- 弹层、菜单、抽屉的进入退出
- 新消息出现时的轻微布局反馈
- 会话切换时消息区域的淡入或布局平滑过渡

不做的事：

- 不给每个元素都加动效
- 不做花哨、夸张、持续干扰阅读的动画
- 简单颜色、阴影过渡仍优先用 CSS 完成

参考：

- Motion for React 官方文档
- https://motion.dev/docs/react

### Playwright MCP

定位：

- 作为本轮前端迁移的页面验证与调试工具
- 用于补充人工浏览器检查，而不是替代设计判断

职责边界：

- 用于检查页面是否真实渲染
- 用于检查关键交互是否仍然可用
- 用于查看控制台错误、页面结构和运行时异常
- 用于辅助判断问题属于样式、动画、资源还是 hydration 层

优先使用场景：

- 每轮 UI 重构后检查首页、登录页、聊天页是否仍然可见
- 检查 `Dialog`、`Sheet`、`DropdownMenu` 等交互型组件是否可正常打开
- 检查输入框、按钮、侧栏、消息区是否因迁移而失效
- 检查移动端或窄屏下结构是否仍然成立
- 在浏览器中出现“空白页 / 元素消失 / 可见性异常”时，辅助判断 DOM 是否实际存在

不做的事：

- 不替代设计语言判断
- 不作为视觉优劣的唯一标准
- 不在当前环境中单独作为唯一事实来源，需要结合真实浏览器观察

当前环境提醒：

- 当前 `Playwright MCP` 可用于访问和调试本项目页面
- 当前环境下可能出现权限提示、字体资源异常或 HMR websocket 噪音
- 因此它更适合做“结构与交互验证”，不应单独承担最终视觉验收

## Untitled UI 参考策略

### 定位

`Untitled UI React` 不是本项目的第二套基础框架，而是本次迁移的重要参考源。

它的价值主要在于：

- 应用层组件组织成熟
- 页面骨架与密度控制稳定
- 表单、侧栏、弹层、空状态、工作区模式完整
- 组件以源代码形式接入，便于学习和局部吸收

### 重点参考范围

优先参考以下模块：

- `Sidebar navigations`
- `Page headers`
- `Modals`
- `Empty states`
- `Loading indicators`
- `Messaging`
- `Command menus`
- `Alerts`
- `Dropdowns`
- `Inputs`
- `Textarea`

### 使用边界

- 参考它的结构、层次、间距、表单与信息组织
- 不直接把 `Untitled UI` 当作新的唯一组件库整套搬入
- 优先吸收其页面模式，而不是盲目复制视觉细节
- 任何引入都要先经过本项目设计语言过滤

### 参考来源

- https://www.untitledui.com/react/docs/introduction
- https://www.untitledui.com/react/components
- https://www.untitledui.com/react/application-ui

## 字体与排版策略

### 字体分工

- 西文正文：`Geist Sans`
- 等宽信息：`Geist Mono`
- 中文正文优先：`MiSans`
- 标题：优先在 `Geist Sans` / `MiSans` 体系内部通过字重、字距、字号和留白建立特征

### 排版原则

- 正文保持克制、紧凑但不拥挤
- 标题比正文更锋利、更明确，但不浮夸
- 页面标题和模块标题层级必须一眼可分
- 代码、时间、状态、模型名等技术信息使用更稳的等宽表达

### 注意事项

- `MiSans` 可商用，但软件中需按官方要求注明使用了 `MiSans`
- 深色模式后续补充时，排版层级不应重做，只做颜色映射

参考：

- Geist Font：https://vercel.com/font
- Geist Design System：https://vercel.com/geist/introduction
- MiSans：https://hyperos.mi.com/font/en/

## 色彩与材质策略

### 主色方向

- 主基调：蓝白
- 辅助色：极浅灰蓝、低饱和冷灰
- 强调色：单一偏科技感的蓝色，不引入第二主强调色

### 渐变策略

- 渐变只承担空气感和空间感
- 主界面避免大面积高饱和渐变
- 优先使用极淡的蓝青白过渡，避免“装饰性过强”的背景

### 表面与边框

- 小圆角
- 弱边框
- 弱阴影
- 少卡片，优先靠留白、层级和分隔建立结构

### 页面材质判断

- 侧栏可以保留略轻的分层感
- 主聊天区应更干净，减少无意义容器感
- 弹层和菜单要比主体略重，但不能喧宾夺主

## 页面结构蓝图

### 整体骨架

- 左侧：可折叠会话导航栏
- 中间：聊天主工作区
- 右侧：暂不固定落地，只保留未来扩展位

### 左侧栏

目标：

- 像高质量开发者产品中的导航面板
- 更轻、更稳、更可收起

包含内容：

- 新建会话
- 会话列表
- 分组与最近活跃
- 用户入口和少量状态信息

### 中间主区

目标：

- 让输入、消息、状态是绝对主角
- 让聊天更接近 `ChatGPT` 的工作区体验

包含内容：

- 顶部标题区
- 消息列表
- 发送输入区
- 错误 / 加载 / 恢复状态

### 右侧扩展位

本轮迁移不作为主交付，但布局应预留：

- 会话元信息
- 上下文摘要
- 模型与模式说明
- 后续功能面板

## 组件迁移顺序

### 第一阶段：基建接入

目标：

- 引入 `Tailwind CSS`
- 初始化 `shadcn/ui`
- 接入 `Lucide React`
- 接入 `Motion`
- 建立字体与 token 基础

产出：

- `components.json`
- 新的 `components/ui` 目录
- 统一的 `cn` 工具
- 统一字体接入方案
- 可开始迁移的新样式体系

### 第二阶段：基础组件替换

目标：

- 替换最常用的基础件，减少旧样式系统对后续页面的牵制

优先迁移：

- `Button`
- `Input`
- `Textarea`
- `Dialog`
- `DropdownMenu`
- `Sheet`
- `Tooltip`
- `Badge`
- `Separator`
- `Skeleton`
- `Alert`

### 第三阶段：结构组件重做

目标：

- 用新设计语言重做关键页面骨架

优先重做：

- `AuthPanel`
- `ConversationSidebar`
- 顶部 `chat-header`
- 空状态
- 加载状态
- 删除 / 重命名确认交互
- `ChatInput`

### 第四阶段：消息工作区精修

目标：

- 统一消息区的排版、层级、留白和状态反馈

处理重点：

- 消息列表密度
- 助手消息与用户消息的区分方式
- 错误消息样式
- 新消息动效
- 滚动到底部和跳转按钮
- 移动端输入区与键盘适配

### 第五阶段：清理与收口

目标：

- 减少旧 CSS 负担
- 清理不再需要的样式类和重复结构
- 为后续深色模式和扩展面板留出口径

## Playwright 验证建议

### 每轮重构后默认检查

- 首页是否可见
- 登录页文案、输入框、按钮是否正常显示
- 聊天工作区骨架是否可见
- 左侧栏是否可展开、收起、切换会话
- 输入区是否可输入、可发送
- 错误提示、加载提示是否能正常出现

### 组件级检查建议

- `Dialog`：打开、关闭、聚焦、遮罩行为
- `Sheet`：移动端或窄屏下的展开关闭
- `DropdownMenu`：触发器、菜单项、关闭行为
- `Textarea`：输入、自动增高、回车发送
- `MessageList`：滚动区、空态、跳转到最新按钮

### 调试判断建议

- 若页面一片空白，但 `Playwright` 仍能读取到 DOM，优先排查：
  - 动画初始状态
  - 可见性样式
  - HMR 未刷新
  - 字体或资源加载异常
- 若 DOM 也不存在，再排查渲染逻辑、运行时错误和服务端响应

输出目标：

输出目标：

- 层级更清楚
- 可折叠
- 移动端可转抽屉
- 视觉上更像导航，不像控制台

### 消息区

主要参考：

- `ChatGPT` 风格的主消息工作区
- 轻气泡、重内容、重可读性

输出目标：

- 助手消息更接近内容流
- 用户消息保留轻气泡或轻背景差异

主要参考：

输出目标：

- 删除确认、重命名、菜单行为统一
- 动效克制
- 键盘与可访问性行为更完整

## 动效策略

### 允许的动效

- 登录页与空态入场
- 侧栏折叠展开
- 弹层与菜单出现消失
- 新消息插入反馈
- 会话切换过渡

### 不建议的动效

- 每条消息都做显著动画
- 高饱和闪烁反馈
- 大面积持续运动背景
- 复杂拖拽或炫技式转场

### 动效标准

- 快
- 稳
- 一致
- 可在移动端顺畅运行
- 对 `prefers-reduced-motion` 友好

## 风险与控制

### 风险一：迁移同时改动业务逻辑过多

控制：

- 样式迁移与业务逻辑拆开
- 单次只改一个区域
- 优先替换结构件，不顺手重写数据流

### 风险二：引入两套设计系统导致风格混乱

控制：

- `shadcn/ui` 只做底座
- `Untitled UI` 只做模式参考
- 统一由项目 token 决定最终风格

### 风险三：页面变精致但聊天主链路变重

控制：

- 聊天主区优先简洁
- 让输入框、消息流和状态反馈成为视觉核心
- 避免把页面重新做成仪表盘拼贴

## 验收标准

### 视觉层

- 页面整体更接近现代开发者产品与 AI 工作台
- 蓝白基调更统一
- 组件风格不再割裂
- 标题、正文、状态信息层级清楚

### 体验层

- 登录更顺畅
- 侧栏更清楚且可折叠
- 消息区更像真实 AI 产品
- 弹层、菜单、加载、错误反馈更完整

### 工程层

- 样式体系从“全局 CSS 主导”转向“token + 组件 + Tailwind”
- 组件边界更清楚
- 后续接深色模式、扩展面板、更多业务组件时摩擦更低

## 当前结论

当前项目已适合进入这次迁移的规划与执行阶段。

推荐执行顺序是：

1. 先接入基建
2. 再替换基础组件
3. 再重做结构组件
4. 最后精修消息工作区

默认不建议一开始就全量推翻现有页面，也不建议在同一轮里同时大改业务逻辑和视觉系统。
