# WebAI Blueprint

## 愿景
构建一个类似 ChatGPT 的网页前端，重点是简洁、美观、易用的聊天体验，并集成 Gemini 服务。

从零搭建一条清晰的新主线，优先保证：
- 聊天体验自然、稳定、现代
- 前后端边界统一，减少跨端数据契约混乱
- 为后续 Supabase 持久化、流式传输、Gemini 生态扩展预留空间

本蓝图面向新的窗口和新的开发周期，不考虑保留原始项目的数据或兼容旧实现。

## 预期技术栈
- 前后端统一采用 TypeScript
- Web 框架采用 Next.js App Router
- 跨端数据校验采用 Zod
- Gemini 接入首版直接放在 Next 服务端
- 持久化方案预留给 Supabase


## Phase 路线

### Phase 1：聊天主链路 MVP
目标：先做出一个可用、好看、结构正确的聊天网页。

包括：
- 聊天页面
- 消息列表
- 输入框与发送按钮
- 加载状态与错误反馈
- Gemini 非流式调用
- 单页内多轮对话

完成标志：
- 用户能发送消息并收到 Gemini 回复
- 页面可以连续对话
- API Key 不暴露给客户端

### Phase 2：流式输出与体验升级
目标：让产品从“能用”升级到“更像聊天产品”。

包括：
- 流式输出
- 增量渲染 assistant 回复
- 中断生成
- 更完整的消息状态
- 自动滚动与输入体验优化

完成标志：
- 回复可以逐步显示
- 用户可以感知更快的交互反馈

### Phase 3：Supabase 持久化与会话系统
目标：让聊天从单页状态变成可保存、可恢复。

包括：
- 接入 Supabase
- conversations / messages 数据模型
- 新建会话
- 读取历史会话
- 恢复历史消息

完成标志：
- 会话列表可见
- 历史消息可恢复

### Phase 4：Gemini 生态扩展
目标：从基础聊天页走向可扩展的 AI 产品。

包括：
- 多模态输入
- 文件理解
- function calling / tools
- structured output
- 搜索或外部知识整合

完成标志：
- 项目具备继续扩展为 AI 工作台的基础

## 界限
- 浏览器只调用 Next 提供的服务端接口
- Gemini API Key 仅存在于服务端
- 所有跨端数据都应通过统一 schema 校验
- 首版先做非流式，流式属于下一阶段
- 首版不优先做用户系统
- 首版不考虑兼容旧工程结构
- 首版不保留原始项目数据
